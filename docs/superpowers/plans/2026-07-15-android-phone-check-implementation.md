# 安卓验机 App 实施计划

## 目标

在 `F:\Code\AndroidPhoneCheck` 实现 Android 10–16 原生离线验机 App。每个阶段都必须保持项目可构建、可测试，并对系统或机型不支持的能力准确降级。

## 阶段 1：工程与开发环境

1. 安装或配置 Android SDK，至少包含 API 29 和 API 36、Build Tools、Platform Tools 及模拟器。
2. 创建 Kotlin、Jetpack Compose、Material 3 单 Activity 工程，设置 `minSdk 29`、`compileSdk 36`、`targetSdk 36`。
3. 加入 Room、Coroutines、Navigation Compose、Lifecycle、CameraX/Biometric 等稳定依赖。
4. 配置 Gradle Wrapper、版本目录、基础代码规范和 `.gitignore`。
5. 建立 debug 构建并运行空白首页测试。

验收：`assembleDebug` 和单元测试可执行，Android 10 与 Android 16 模拟器能启动应用。

## 阶段 2：领域模型与本地状态

1. 定义检测分类、检测项目、状态、严重程度、能力状态、原始数据和用户确认结果。
2. 定义统一 `DiagnosticModule` 接口，包含能力探测、启动、停止和结果流。
3. 建立 Room 实体、DAO、数据库及领域模型映射。
4. 实现新建会话、保存进度、继续验机、跳过、重测和清除会话。
5. 为状态转换、严重程度排序、持久化恢复编写单元测试。

验收：进程重启后能恢复同一验机会话；状态与数据库记录一致。

## 阶段 3：主流程与结果界面

1. 实现首页、项目总览、单项测试容器和结果汇总页。
2. 实现按模块顺序导航，同时允许用户返回、跳过和重测。
3. 统一展示正常、异常、风险提示、未测试、不支持和权限不足。
4. 汇总页按严重程度置顶异常，并显示异常、风险和未完成数量。
5. 为跳过、重测、退出恢复和汇总排序编写 Compose 测试。

验收：使用模拟检测模块可从首页完整走到汇总页，退出后可继续。

## 阶段 4：低风险自动检测

1. 基本信息：Build、系统版本、ABI、内存、存储、显示参数。
2. 电池：电量、温度、电压、充电状态和充电来源。
3. 传感器清单：枚举可用传感器并采集实时变化。
4. 能力清单：相机数量、闪光灯、NFC、蓝牙、Wi-Fi、定位、生物识别和 USB Host 能力。
5. 存储：应用私有目录读写自检。

验收：缺失硬件或 API 返回空值时不崩溃，结果正确显示“不支持”。

## 阶段 5：屏幕、触摸与实体功能

1. 实现黑、白、红、绿、蓝、灰阶全屏观察流程。
2. 实现覆盖网格划线测试、断点提示、边缘区域和多点触控显示。
3. 实现亮度测试并在退出时恢复原亮度。
4. 实现音量键事件、震动和手电筒测试；电源键仅允许人工确认。
5. 确保生命周期中断时释放资源并保存测试进度。

验收：触控轨迹可视化准确，未覆盖区域清晰，页面退出不遗留亮度或闪光灯状态。

## 阶段 6：摄像头与音频

1. 使用公开 API 枚举可用摄像头能力，不把逻辑镜头数量等同于物理镜头数量。
2. 实现前后摄预览、切换、对焦、闪光灯、拍照和短视频测试。
3. 实现扬声器测试音、麦克风录音波形和即时回放。
4. 听筒、耳机和蓝牙音频无法稳定自动路由时采用人工确认，并记录测试条件。
5. 处理摄像头或麦克风被占用、权限拒绝和资源初始化失败。

验收：权限拒绝不阻断其他项目；离开页面后摄像头和录音资源完全释放。

## 阶段 7：连接、定位与生物识别

1. 按 Android 版本封装蓝牙、附近设备、Wi-Fi 和定位权限差异。
2. 检测硬件能力及系统开关状态，不擅自修改系统设置。
3. 实现定位获取测试，并显示精度和等待超时。
4. 实现 NFC 能力和启用状态检查。
5. 使用 BiometricPrompt 调起系统认证，不读取或保存生物信息。

验收：授权、拒绝、永久拒绝、关闭系统开关和硬件缺失均有不同且准确的结果。

## 阶段 8：安全风险提示

1. 聚合公开可见的 Root、测试签名、可疑文件、调试构建和模拟器线索。
2. 每条线索独立记录证据，不输出“绝对 Root”或“必定翻新”。
3. 将 Bootloader 和系统完整性中无法由普通离线 App 可靠判断的项目标记为不支持。
4. 为误报边界编写测试和说明文案。

验收：所有安全判断均可追溯到具体线索，汇总页只显示风险提示。

## 阶段 9：兼容性与发布准备

1. 在 Android 10–16 模拟器执行构建、安装、旋转、后台恢复和权限矩阵测试。
2. 建立小米、华为或荣耀、OPPO 或 vivo、三星真机验收表。
3. 修复崩溃、ANR、资源泄漏和错误结论，记录尚未验证的品牌能力。
4. 完善隐私说明、权限用途说明、应用图标、版本号和签名配置说明。
5. 生成 debug APK；发布签名由项目所有者保管，不提交仓库。

验收：自动化测试通过；完整流程无阻断；任何单项失败不导致应用退出；首版边界与设计文档一致。

## 实施顺序

优先完成阶段 1–4，交付可运行的基础版本；随后逐项加入需要真机交互的阶段 5–8；最后集中完成兼容性验证。每个模块单独提交，避免把大量未经验证的硬件代码混入一次提交。
